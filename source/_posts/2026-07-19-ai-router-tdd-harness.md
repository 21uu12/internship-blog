AI 应用很容易陷入一种开发方式：改几句 Prompt，手工试两个问题，感觉结果不错就结束。但只要输入换一种说法，或者真实 API 出现超时、限流和格式异常，系统就会暴露出大量没有被定义的行为。

这次我用 TypeScript 和 Vitest 做了一个小型 AI 任务路由器，用它练习两套互补的方法：确定性代码使用 TDD，概率性模型使用评估 Harness。

## 1. 路由器要解决什么问题

路由器接收用户输入，并在四个标签中选择一个：

- `summarizer`：总结内容；
- `code_explainer`：解释代码或分析错误；
- `task_planner`：拆解和规划任务；
- `unknown`：不属于以上任务或无法可靠判断。

第一版是关键词规则。它简单、便宜、可预测，但“总结”“概括”“浓缩”“TL;DR”都会增加规则维护成本，混合语言和否定表达也容易误判。

后来加入 LLM，不是为了删除测试，而是为了把测试从单个函数扩展到模型行为。

## 2. TDD 管确定性代码

TDD 的循环是 Red、Green、Refactor。

### Red：先写失败测试

先写一个业务测试，然后运行它，确认测试因为缺少新行为而失败。

Red 必须真的运行。否则无法证明测试能检测到缺失行为，也无法区分业务失败和导入、配置等无关错误。

### Green：最少代码通过

只增加足以让当前测试通过的实现。第一轮甚至可以硬编码，因为 Green 的目标不是立即设计出最终架构，而是尽快建立有效反馈。

### Refactor：保持全绿再整理结构

测试通过后再提取规则表、参数化案例、消除重复。重构只改变代码结构，不改变已经被测试锁定的行为。

## 3. Harness 管概率性行为

Harness 可以理解为模型的试验台。它由三部分组成：

1. 带标准答案的评估案例；
2. 批量执行案例的 Evaluator；
3. 被评估的 Router。

```text
评估案例 -> Evaluator -> Router -> 模型客户端 -> API
                 |
                 -> 通过数、准确率、失败案例
```

它之所以能自动算分，是因为每个案例都有 `input` 和 `expected`。Evaluator 调用路由器取得 `actual`，比较两者并汇总结果。

准确率只是一个入口，真正有价值的是失败案例。失败案例会推动我们定义新的产品边界，再把这个边界写进 Prompt 和回归案例。

## 4. 把 LLM 放在可测试的边界后面

模型调用被封装在一个很小的接口后面：

```ts
interface ModelClient {
  generate(prompt: string): Promise<string>;
}
```

单元测试注入 Fake Model，因此不需要联网，也不会消耗额度。真实供应商只在独立的冒烟测试和完整评估中使用。

LLM 返回值还要经过 JSON 解析和标签白名单校验。格式错误或非法标签统一降级到 `unknown`。

这个边界带来三个直接收益：

1. 可以更换模型供应商；
2. 可以稳定测试 Prompt 拼接和输出解析；
3. 可以把网络问题与业务逻辑问题分开诊断。

## 5. 从失败案例推动 Prompt V2

评估集从 9 个扩展到 30 个案例后，规则基线在该集合上的结果从 `6/9` 变成 `14/30`。

这不代表代码突然变差了，而是更丰富的案例暴露了原先没有测到的边界。

LLM + Prompt V1 在这 30 个案例上通过了 29 个。唯一失败的输入同时包含英文动作和中文内容，大意是：

```text
总结这段 TypeScript 代码
```

模型看见“代码”后选择了代码解释，却忽略了更明确的动作“总结”。

因此 Prompt V2 没有简单添加一个特殊示例，而是定义了更通用的决策规则：

1. 明确请求的动作优先于内容类型；
2. 内容类型只作为辅助信号；
3. 被否定的动作不能作为意图；
4. 多个动作并存且无法确定优先级时返回 `unknown`。

V2 已通过原失败案例的定向验证。但完整的 30 案例回归尚未记录，因此还不能声称它已经达到 `30/30`。

## 6. 三种测试不能混为一谈

### 单元测试

使用假模型验证本地代码契约。它应该快速、离线、确定，适合每次修改后运行。

### 冒烟测试

使用少量真实请求，确认鉴权、网络、模型和解析链路能完整走通。冒烟测试通过只代表系统“能跑”，不代表模型“足够准”。

### 完整评估

批量运行更多真实案例，用来比较 Prompt 或模型版本。即使在 30 个案例上取得高准确率，也只能说明该小型评估集的结果，不能代表真实世界的普遍质量。

## 7. 真实模型 API 的工程问题

接入真实模型后，常见问题不只是 TypeScript 语法，还包括密钥、代理、超时和配额。

- 密钥只放环境变量或密钥管理服务，不写入源码、日志和文章；已经暴露的密钥必须撤销。
- 为单请求设置超时，只对适合重试的错误做有限次数的指数退避。
- 先跑小型 smoke 集，再决定是否运行完整评估。
- 异步不等于并行。循环中逐项 `await` 仍是串行，速度较慢，但更容易控制限流。
- 切换供应商时重新检查代理配置，不要默认复用上一家服务的网络设置。

## 8. 可复用的 AI Native 开发循环

确定性代码继续使用 TDD：

```text
Red -> Green -> Refactor
```

概率性模型使用评估循环：

```text
dataset
  -> harness
  -> failure analysis
  -> product policy
  -> prompt version
  -> targeted check
  -> full regression
```

这两套循环并不冲突。TDD 保证本地代码契约，Harness 衡量模型在一组代表性输入上的行为。

## 9. 复用检查清单

- [ ] 标签是否有限、互斥，并有安全的 `unknown`？
- [ ] 每个确定性规则是否先有一个会正确失败的测试？
- [ ] 模型 SDK 是否被小接口隔离？
- [ ] 模型输出是否经过结构化解析和白名单校验？
- [ ] 日常测试是否离线、快速、确定？
- [ ] 是否先跑 smoke，再跑完整评估？
- [ ] Prompt 修改是否来自明确的失败模式？
- [ ] 新 Prompt 是否先定向验证，再完整回归？
- [ ] 是否避免在源码、日志和文章中保存密钥？
- [ ] 是否把小样本结果严格描述为“该评估集上的结果”？

## 10. 下一步

下一次实验先完成 Prompt V2 的 30 案例完整回归，然后补充“否定、多意图、内容类型干扰”的成组案例，并按标签统计召回率或混淆矩阵。

这样才能发现总准确率背后，某一类路由是否存在持续性的系统误判。
